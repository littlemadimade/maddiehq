# Deploy AppSeed on AWS EC2

Single-instance deployment on EC2 with Docker Compose, Caddy for HTTPS, and optional Litestream backups to S3. Estimated cost: **$8-17/month** (t3.micro or t3.small).

---

## Cost Breakdown

| Resource | Monthly Cost |
|---|---|
| t3.micro (1 vCPU, 1 GB) | ~$8.50 (or free-tier eligible for 12 months) |
| t3.small (2 vCPU, 2 GB) | ~$17.00 |
| 20 GB gp3 EBS | ~$1.60 |
| Elastic IP (while attached) | $0.00 |
| Data transfer (first 100 GB) | ~$0.00–$9.00 |

For most SaaS apps starting out, **t3.micro** is sufficient. Upgrade to **t3.small** when you need more memory for larger SQLite databases or heavier traffic.

## Quick Start (Console)

### 1. Launch an EC2 instance

1. Go to **EC2 > Launch Instance** in the AWS Console
2. Choose **Amazon Linux 2023** AMI (x86_64)
3. Select **t3.micro** instance type
4. Create or select an SSH key pair
5. Under **Network settings**, create a security group with these rules:

| Type | Port | Source | Purpose |
|---|---|---|---|
| SSH | 22 | Your IP (`x.x.x.x/32`) | SSH access |
| HTTP | 80 | `0.0.0.0/0` | Web traffic |
| HTTPS | 443 | `0.0.0.0/0` | SSL termination |

6. Under **Configure storage**, add a second EBS volume:
   - Size: **20 GB**
   - Type: **gp3**
   - Device: `/dev/xvdf`
   - **Do not** check "Delete on termination"

7. Under **Advanced details > User data**, paste the contents of [`deploy/ec2/user-data.sh`](../../deploy/ec2/user-data.sh) — editing `APP_REPO`, `APP_BRANCH`, and `DOMAIN` at the top

8. Launch the instance

### 2. Allocate an Elastic IP

1. Go to **EC2 > Elastic IPs > Allocate Elastic IP address**
2. Associate it with your instance
3. Point your DNS A record to this IP

### 3. Configure environment variables

SSH into the instance and edit the `.env` file:

```bash
ssh -i your-key.pem ec2-user@YOUR_ELASTIC_IP
sudo vi /opt/appseed/.env
```

Set all required variables:

```env
BETTER_AUTH_SECRET=<auto-generated-by-user-data>
BETTER_AUTH_URL=https://yourdomain.com
APP_URL=https://yourdomain.com
APP_NAME=YourApp

# OAuth (optional)
GOOGLE_CLIENT_ID=
GOOGLE_CLIENT_SECRET=
GITHUB_CLIENT_ID=
GITHUB_CLIENT_SECRET=

# Email
RESEND_API_KEY=re_...

# Stripe
STRIPE_SECRET_KEY=sk_live_...
STRIPE_PRICE_ID=price_...
STRIPE_WEBHOOK_SECRET=whsec_...
```

Restart the app after editing:

```bash
sudo systemctl restart appseed
```

## Quick Start (CloudFormation)

For a fully automated setup, use the included CloudFormation template:

```bash
aws cloudformation create-stack \
  --stack-name appseed \
  --template-body file://deploy/ec2/cloudformation.yml \
  --capabilities CAPABILITY_NAMED_IAM \
  --parameters \
    ParameterKey=KeyName,ParameterValue=your-keypair \
    ParameterKey=AppRepo,ParameterValue=https://github.com/you/your-app.git \
    ParameterKey=Domain,ParameterValue=app.example.com \
    ParameterKey=SSHCidr,ParameterValue=$(curl -s ifconfig.me)/32 \
    ParameterKey=LitestreamBucket,ParameterValue=my-backup-bucket
```

This creates:
- VPC with public subnet, internet gateway, and route table
- Security group (SSH, HTTP, HTTPS)
- EC2 instance with Docker + Docker Compose
- 20 GB encrypted gp3 EBS data volume
- Elastic IP
- IAM role with SSM access + optional S3 permissions for Litestream

Check progress:

```bash
aws cloudformation describe-stack-events --stack-name appseed --query 'StackEvents[0:5]'
```

Get the Elastic IP once complete:

```bash
aws cloudformation describe-stacks --stack-name appseed \
  --query 'Stacks[0].Outputs[?OutputKey==`PublicIP`].OutputValue' --output text
```

## SSL/HTTPS Options

### Option A: Caddy (Recommended for single-instance)

Caddy is the simplest approach — automatic HTTPS via Let's Encrypt with zero configuration.

If you set the `DOMAIN` variable in `user-data.sh` (or the CloudFormation `Domain` parameter), Caddy is installed and configured automatically. The Caddyfile reverse-proxies to your app on port 3013.

**Manual setup** (if you skipped it during launch):

```bash
# Amazon Linux 2023
sudo dnf copr enable -y @caddy/caddy
sudo dnf install -y caddy

# Create Caddyfile
sudo tee /etc/caddy/Caddyfile > /dev/null <<'EOF'
app.example.com {
    reverse_proxy localhost:3013
}
EOF

sudo systemctl enable --now caddy
```

Caddy automatically obtains and renews certificates. Ensure port 80 is open (required for ACME challenges).

### Option B: AWS Application Load Balancer (ALB)

Use an ALB if you want AWS-managed SSL via ACM, or plan to scale to multiple instances later.

1. **Request a certificate** in AWS Certificate Manager (ACM) for your domain
2. **Create a target group** pointing to your instance on port 3013
3. **Create an ALB** with:
   - Listener on port 443 (HTTPS) using the ACM certificate
   - Listener on port 80 redirecting to HTTPS
   - Forward to your target group
4. Point your DNS to the ALB's DNS name (CNAME or Route 53 alias)

**Cost note:** An ALB adds ~$18/month, which doubles the cost of a micro instance. Use Caddy unless you specifically need ALB features (multi-instance, WAF integration, etc.).

### Option C: Nginx + Certbot

```bash
sudo dnf install -y nginx certbot python3-certbot-nginx  # Amazon Linux 2023
# or: sudo apt install -y nginx certbot python3-certbot-nginx  # Ubuntu

sudo tee /etc/nginx/conf.d/appseed.conf > /dev/null <<'EOF'
server {
    listen 80;
    server_name app.example.com;
    location / {
        proxy_pass http://localhost:3013;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
EOF

sudo nginx -t && sudo systemctl enable --now nginx
sudo certbot --nginx -d app.example.com --non-interactive --agree-tos -m you@example.com
```

## EBS Volumes

The deployment uses two EBS volumes:

| Volume | Mount | Purpose |
|---|---|---|
| Root (`/dev/xvda`) | `/` | OS + Docker images (20 GB gp3) |
| Data (`/dev/xvdf`) | `/data` | SQLite database + Docker volumes (20 GB gp3) |

The data volume has `DeletionPolicy: Snapshot` in CloudFormation, meaning AWS takes a snapshot before deleting it if the stack is torn down.

### Manual snapshots

```bash
# Find the volume ID
VOLUME_ID=$(aws ec2 describe-volumes \
  --filters "Name=tag:Name,Values=appseed-data" \
  --query 'Volumes[0].VolumeId' --output text)

# Create a snapshot
aws ec2 create-snapshot \
  --volume-id "$VOLUME_ID" \
  --description "AppSeed data backup $(date +%Y-%m-%d)"
```

### Automated snapshots

Use [Amazon Data Lifecycle Manager](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/snapshot-lifecycle.html) to create daily snapshots with a 7-day retention policy. This costs ~$0.05/GB-month for stored snapshots.

## Litestream Backups to S3

Litestream continuously replicates your SQLite database to S3, giving you point-in-time recovery.

### 1. Create an S3 bucket

```bash
aws s3 mb s3://your-app-backups --region us-east-1
```

### 2. IAM permissions

If you used the CloudFormation template with `LitestreamBucket`, permissions are already configured. Otherwise, create a policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:PutObject", "s3:DeleteObject", "s3:ListBucket"],
      "Resource": [
        "arn:aws:s3:::your-app-backups",
        "arn:aws:s3:::your-app-backups/*"
      ]
    }
  ]
}
```

Attach this policy to your EC2 instance role.

**Using an instance role (recommended):** If the EC2 instance has an IAM role with S3 permissions, Litestream picks it up automatically — no access keys needed. Set these in `.env`:

```env
LITESTREAM_REPLICA_BUCKET=your-app-backups
LITESTREAM_REPLICA_PATH=db
LITESTREAM_REPLICA_REGION=us-east-1
# LITESTREAM_REPLICA_ENDPOINT is not needed for native S3
# Omit LITESTREAM_ACCESS_KEY_ID and LITESTREAM_SECRET_ACCESS_KEY to use the instance role
```

**Using access keys (not recommended):** If you must use static credentials:

```env
LITESTREAM_ACCESS_KEY_ID=AKIA...
LITESTREAM_SECRET_ACCESS_KEY=...
LITESTREAM_REPLICA_BUCKET=your-app-backups
LITESTREAM_REPLICA_PATH=db
LITESTREAM_REPLICA_REGION=us-east-1
```

### 3. Enable Litestream in Docker Compose

Litestream is already defined in `docker-compose.yml` under the `prod` profile. It starts automatically with `docker compose --profile prod up`.

### 4. Restore from backup

```bash
# Stop the app
sudo systemctl stop appseed

# Restore the database
docker run --rm \
  -v appseed_node_data:/data \
  -e LITESTREAM_ACCESS_KEY_ID=... \
  -e LITESTREAM_SECRET_ACCESS_KEY=... \
  litestream/litestream:latest restore \
  -o /data/appseed.db \
  s3://your-app-backups/db

# Restart
sudo systemctl start appseed
```

## Systemd Service

The `appseed.service` file ensures Docker Compose starts on boot and restarts on failure.

```bash
# View logs
sudo journalctl -u appseed -f

# Restart
sudo systemctl restart appseed

# Check status
sudo systemctl status appseed

# Rebuild and restart (after git pull)
cd /opt/appseed && git pull
sudo systemctl reload appseed
```

```bash
ssh -i your-key.pem ec2-user@YOUR_ELASTIC_IP
```

## Monitoring

### CloudWatch (basic)

EC2 instances automatically report CPU, disk, and network metrics to CloudWatch. Enable **detailed monitoring** ($2.10/month) for 1-minute granularity.

### Health check

AppSeed includes a health endpoint:

```bash
curl http://localhost:3013/api/health
# {"ok":true,"db":true,"timestamp":"2026-03-23T..."}
```

Set up a CloudWatch alarm or use an external service (UptimeRobot, Better Uptime) to monitor this endpoint.

### Docker logs

```bash
# All containers
docker compose --profile prod logs -f

# Just the app
docker compose --profile prod logs -f node-prod

# Just Litestream
docker compose --profile prod logs -f litestream
```

## Security Checklist

- [ ] SSH restricted to your IP only (not `0.0.0.0/0`)
- [ ] SSH key pair stored securely; root login disabled
- [ ] HTTPS enabled via Caddy, ALB, or Nginx+Certbot
- [ ] `.env` file has restrictive permissions (`chmod 600`)
- [ ] `BETTER_AUTH_SECRET` is a unique 32+ character random string
- [ ] EBS data volume is encrypted
- [ ] Automatic security updates enabled (`dnf-automatic` or `unattended-upgrades`)
- [ ] Litestream backups configured and verified
- [ ] IAM role follows least-privilege (only S3 access needed for Litestream)
- [ ] Stripe webhook secret is set (without it, the webhook route returns 500)

### Enable automatic security updates

**Amazon Linux 2023:**

```bash
sudo dnf install -y dnf-automatic
sudo systemctl enable --now dnf-automatic-install.timer
```

**Ubuntu:**

```bash
sudo apt install -y unattended-upgrades
sudo dpkg-reconfigure -plow unattended-upgrades
```

## Troubleshooting

### Cloud-init logs

```bash
# Full user-data execution log
cat /var/log/appseed-init.log

# Cloud-init system log
cat /var/log/cloud-init-output.log
```

### Docker won't start

```bash
sudo systemctl status docker
sudo journalctl -u docker -n 50
```

### App container keeps restarting

```bash
docker compose --profile prod logs node-prod --tail 50

# Common causes:
# - Missing BETTER_AUTH_SECRET → generate with: openssl rand -base64 32
# - Port conflict → check: sudo ss -tlnp | grep 3013
# - Build failure → rebuild: DOCKER_BUILDKIT=0 docker compose --profile prod build node-prod
```

### EBS volume not mounted

```bash
lsblk                      # Check if the device is visible
sudo blkid /dev/xvdf       # Check filesystem
cat /etc/fstab              # Verify fstab entry
sudo mount -a               # Attempt mount
```

### Caddy certificate issues

```bash
sudo systemctl status caddy
sudo journalctl -u caddy -n 50

# Common causes:
# - Port 80 not open in security group (required for ACME)
# - DNS not pointing to the Elastic IP yet
# - Rate limit hit (Let's Encrypt allows 5 certs per domain per week)
```
